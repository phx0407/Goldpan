## From DEC000002 (Restaurant Update Submission) — final draft (v4.1), ready for approval

v4 was approved architecturally; v4.1 is a mechanical-and-integrity correction pass (event actors split into `actor_type`/`actor_id`, mirroring DEC000001; the resubmission-chain cycle-prevention claim corrected — branching factor ≤ 1 alone does not prove acyclicity, so an explicit ancestry check or `chain_depth` invariant is now specified; child-resubmission creation formalized as a discrete `submission.restaurant_update.resubmit` command; each proposed downstream foreign key labeled by implementation-status maturity instead of presented uniformly as ready; the "one downstream object per disposition" contradiction between §5.4 and §6 resolved; `failure_stage` added to the `failed` disposition model; downstream-completion authority specified precisely; `no_action` approval tightened to require `resolution_summary`; `returned`-parent archival eligibility clarified; several miscited section references corrected) that changes no policy choice. The nine Founder decisions below (DEC000002 §7) are unchanged in substance from v4 and remain the complete, final list — the only additions are the new `resubmit` command and the FK-maturity labeling, both flagged explicitly within decisions #14 and #16 below.

10. **Approve the five review statuses:** `pending_review`, `in_review`, `returned`, `approved`, `rejected`.
11. **Approve the separate disposition model and its five statuses** (`unassessed`, `pending`, `in_progress`, `completed`, `failed`, with `failure_stage` — added in v4.1 — sub-classifying `failed` into `handoff_call`/`local_write`/`downstream_terminal`), with routing completion distinguished from downstream-work completion and completion authority specified precisely (DEC000002 §5.3).
12. **Approve the four disposition types and their one-owner routing boundaries** — in particular, that `exception_escalation` is owned solely by Governance OS (not jointly with Knowledge OS), that identity/contact changes route to the Blueprint's existing Identity Review Queue rather than being edited directly by the submission reviewer, and that `no_action` now requires a mandatory `resolution_summary` (added in v4.1, since it is the only record that disposition ever produces) (DEC000002 §5.2, §5.4).
13. **Approve immutable parent submissions with linked child resubmissions**, including the v4.1-corrected cycle-prevention mechanism — an ancestry check at resubmission time (or a `chain_depth` invariant), since the v4 claim that branching factor ≤ 1 alone proves acyclicity was incorrect and has been withdrawn (DEC000002 §5.7-§5.8).
14. **Approve splitting the broad review command into `claim`/`release`/`return`/`approve`/`reject`, removing `submission.close_no_action` as a standalone command, and adding `submission.restaurant_update.resubmit` as a newly formalized discrete command** (new in v4.1 — previously only described narratively) (DEC000002 §5.5, §5.7).
15. **Approve archival as a separate attribute set**, with the eligibility rules in DEC000002 §5.9, including the v4.1 clarification that a `returned` parent's archival eligibility depends only on having a linked child, not on that child's own outcome.
16. **Choose the downstream-link storage model:** explicit nullable foreign-key fields (recommended, DEC000002 §5.11) or a typed generic downstream-link table — and approve labeling each of the four candidate FK columns by implementation-status maturity (existing target confirmed / target entity unresolved / Blueprint-defined-not-implemented / future recommendation with no confirmed target), added in v4.1 in place of presenting all four as uniformly build-ready.
17. **Confirm the exact source of stable user identity** for `claimed_by_user_id`/`archived_by_user_id` — same open question as DEC000001 decision 9, shared across both records.
18. **Confirm that `operations.partner_submissions` remains outside the scope of this decision** — no changes proposed to it.
