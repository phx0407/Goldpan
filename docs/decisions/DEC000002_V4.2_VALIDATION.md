## Seven corrections — confirmed

**1. Ineffective cycle-prevention rule replaced.** §5.8 withdraws the v4.1 ancestry-ID check as vacuous (a freshly generated child ID can never already appear in an existing chain, so the check always passed and caught nothing) and replaces it with the structural guarantee: `resubmit` always creates a brand-new row, never attaches an existing row as child, `resubmission_of_submission_id`/`superseded_by_submission_id` are immutable post-creation, no routine command may rewire a chain, and parent-child linkage is created atomically. Existing/imported rows (bulk import, migration, manual correction) are flagged as needing a separate validation mechanism, not covered by this construction argument. The inaccurate cross-row `CHECK`-constraint example is removed; §5.8 now states a trigger, deferred constraint trigger, or application-level transactional validation is required for any cross-row chain-depth invariant. §7 item 4, §8 Guardrails, §8 Validation criteria, and §8 Revisit triggers were all updated to stop referencing "the ancestry check" and instead cite the construction guarantee.

**2. DEC000001 dependency claim corrected.** §5.5, §5.11, and the header revision note no longer attribute the packet-vs-session handoff-target choice to "DEC000001's own open question." §5.11 documents the schema inspection performed (migration 015 confirms `operations.intake_packets` as the only real table; no `intake_session` table exists anywhere; migration 011's `resulting_intake_session` is a free-text, non-FK placeholder per its own inline comment) and resolves `operations.intake_packets` as the canonical downstream object, with `resulting_intake_session_id` demoted to an optional, non-canonical secondary reference. No new DEC000003 was needed — the question was resolvable directly from evidence already in the repo, not genuinely open.

**3. Downstream-link cardinality corrected.** §5.11 replaces "exactly one of four FK columns is always populated" with the conditional table: zero canonical FKs for `no_action`; zero for any routed disposition still `pending`; exactly one canonical FK once a handoff succeeds (`in_progress`/`completed`); zero for `failed`/`handoff_call`; the downstream object may exist without its local FK recorded during `failed`/`local_write` reconciliation; and no submission may have more than one canonical FK populated simultaneously. §6's code-block comments, §7 item 7, §8 Guardrails, and §8 Validation criteria were all updated to match — `resulting_intake_session_id` is explicitly excluded from the cardinality count throughout.

**4. `local_write` failure made recordable.** §5.3's `failure_stage` definition and §5.12 (split into 3a "Recording the failure" and 3b "Retrying safely") now specify that when a downstream call succeeds but the local linkage transaction fails, a separate recovery transaction — not the failed transaction itself — must record `disposition_status = failed`, `failure_stage = local_write`, and attempt metadata. §8 Guardrails and Validation criteria each gained a new bullet requiring this; §8 Risks gained a bullet flagging that detection of the local-write failure itself is an implementation-level monitoring gap.

**5. Initiating and executing actors preserved.** §5.10's event schema adds `initiating_actor_type`, `initiating_actor_id`, and `downstream_caller_id` alongside the original `actor_type`/`actor_id`. The accompanying prose specifies initiating-actor fields are populated on handoff events triggered by a human, left null for pure human actions where `actor_id` already answers both questions, and describes `downstream_caller_id` as capturing the downstream owning OS's own callback identity on `downstream_completion_received` events. §8 Guardrails and Validation criteria each gained a bullet requiring this population.

**6. `resubmit` invocation authority explicitly deferred.** §5.7's Actor bullet, §7 item 5, and §8 Guardrails now state that Founder approval of `resubmit`'s existence/design does not promote its registry status past `draft`; it stays `draft` until a separate role/portal-origin decision resolves who may invoke it. No vague "existing submission-origination model" language remains.

**7. Approval-history wording tightened.** The header's Status and Revision-note lines now read "v4 was accepted in direction during review — not Founder-approved as architecture," replacing "v4 is approved architecturally." The record's own status remains `final draft — ready for Founder approval`.
