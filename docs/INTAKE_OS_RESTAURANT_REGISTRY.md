# GoldPan Intake OS — Restaurant Registry
**Version:** 1.0  
**Date:** 2026-07-03  
**Scope:** Defines the Restaurant_Registry and Lifecycle_Events schemas — the master record of every restaurant in the GoldPan system and the audit trail of every lifecycle transition

---

## Purpose

The Restaurant_Registry is the master record for every restaurant GoldPan has ever evaluated. It answers, at a glance: **what is the current state of this restaurant's knowledge, and what is the next required action?**

The Restaurant_Registry does not store evidence. Evidence lives in the GDL (dish rows, ingredient rows, allergen disclosures). The Registry stores lifecycle state, coverage rationale, and operational tracking — the administrative layer above the evidence system.

---

## The Two Components

The Restaurant_Registry consists of two database tabs:

**Tab 1 — `Restaurant_Registry`**: One row per restaurant. Current state only. Updated in place as the restaurant moves through the lifecycle.

**Tab 2 — `Lifecycle_Events`**: One row per lifecycle transition. Append-only. The permanent audit trail.

These two tables together provide both a current snapshot (Registry) and a complete history (Events) for every restaurant in GoldPan.

## Restaurant_Registry Schema

### Core Identity

| Column | Type | Description |
|--------|------|-------------|
| `Restaurant_ID` | Text | Unique identifier generated by `next_id.py`. Never typed manually. Format: `rest_XXXX`. |
| `Name` | Text | Restaurant name as publicly known |
| `Address` | Text | Street address |
| `Neighborhood` | Text | Neighborhood or district (used for geographic filtering) |
| `City` | Text | City |
| `State` | Text | State or province |

### Prospect Tracking

| Column | Type | Description |
|--------|------|-------------|
| `Prospect_Source` | Enum | How the restaurant was identified. Values: `customer_request` / `canvasser_discovery` / `market_expansion` / `partner_referral` |
| `Prospect_Date` | Date | Date the restaurant was added as a prospect |
| `Prospect_Notes` | Text | Brief context for why this restaurant was surfaced (e.g., "Customer requested via contact form" or "Found during Pilsen canvassing run") |

### Lifecycle State

| Column | Type | Description |
|--------|------|-------------|
| `Lifecycle_Status` | Enum | Current stage. Values: `prospect` / `qualified` / `onboarding` / `evidence_acquisition` / `verification` / `qa_review` / `published` / `recanvassing` / `suspended` / `deactivated` |
| `Status_Updated` | Date | Date of last lifecycle status change |
| `Status_Updated_By` | Text | Actor who made the last status change (canvasser name, reviewer name, or `pipeline`) |

### Coverage

| Column | Type | Description |
|--------|------|-------------|
| `Coverage_Signal` | Text | The documented rationale for coverage approval. Which signal dimension(s) justified coverage, and what specific evidence supports that assessment. | `Coverage_Approved_By` | Text | Name of person who approved coverage (prospect → qualified) |
| `Coverage_Approved_Date` | Date | Date coverage was approved |

### Evidence

| Column | Type | Description |
|--------|------|-------------|
| `Evidence_Tier` | Enum | Quality level of the restaurant's dietary evidence. See Evidence Tier definitions below. Values: `Tier_1_Confirmed` / `Tier_2_Disclosed` / `Tier_3_Inferred` |
| `Has_Allergen_Guide` | Boolean | TRUE if the restaurant publishes an allergen guide (formal or informal) |
| `Primary_Source_URL` | URL | The primary menu source URL |
| `Primary_Source_Tier` | Enum | Source quality tier. Values: `Tier_1` / `Tier_2` / `Tier_3`. See Source Tier definitions below. |

### Operations

| Column | Type | Description |
|--------|------|-------------|
| `Canvasser` | Text | Assigned canvasser name |
| `QA_Reviewer` | Text | Assigned QA reviewer name |
| `Last_Canvassed` | Date | Date evidence was last verified or confirmed current |
| `Published_Date` | Date | Date the restaurant was first published in GoldPan (first `qa_review → published` transition) |
| `Recanvass_Status` | Enum | Freshness state for published restaurants. Values: `current` / `due_soon` / `overdue` / `needs_review`. See RECANVASSING_POLICY.md. |

### Exceptions

| Column | Type | Description |
|--------|------|-------------|
| `Suspension_Reason` | Text | If `suspended`: brief reason (e.g., "Restaurant temporarily closed for renovation" or "Website down — menu inaccessible") |
| `Deactivation_Reason` | Text | If `deactivated`: brief reason (e.g., "Restaurant permanently closed as of 2026-05-01") |
| `Notes` | Text | Freeform field for anything not captured above |

## Evidence Tier Definitions

Evidence tiers describe the reliability of the dietary conclusions GoldPan can draw for a restaurant's dishes. Tier is determined by the quality and sourcing of dietary information — not by the size of the menu.

**Evidence Tier 1 — Confirmed**  
The restaurant has directly provided or confirmed allergen or ingredient information through a primary-source interaction: a confirmation call, email exchange, staff interview, or provision of an internal allergen guide not otherwise published.

Tier 1 evidence carries the highest confidence. GoldPan can make specific, named allergen conclusions with `confirmed` status.

**Evidence Tier 2 — Disclosed**  
The restaurant publicly publishes allergen or ingredient information on its own website, menu PDF, or official materials. The restaurant has disclosed — the canvasser has documented what was disclosed. No direct confirmation interaction is required.

Tier 2 evidence carries high confidence where the disclosure is clear and specific.

**Evidence Tier 3 — Inferred**  
Dietary conclusions are drawn from menu language and ingredient inference without explicit allergen or ingredient disclosure. The canvasser uses available menu language, preparation descriptions, and ingredient knowledge to make dietary assessments.

Tier 3 evidence requires careful governing-language treatment in the product: conclusions are `inferred`, not `confirmed`. Users are informed of the evidence basis.

> **Note on tier progression:** Evidence Tier is a property of the restaurant's current best evidence, not a permanent rating. A restaurant may begin at Tier 3 and advance to Tier 2 when the restaurant publishes an allergen guide, or to Tier 1 when GoldPan completes a confirmation interaction. Tier progression should trigger a recanvassing pass to update dish records.

## Source Tier Definitions

Source tiers describe where the menu information came from. Source tier is a property of individual sources in the Menu Source Registry, not of the restaurant overall. A restaurant may have multiple sources at different tiers.

The `Primary_Source_Tier` in the Restaurant_Registry reflects the tier of the restaurant's primary active source.

**Source Tier 1 — Restaurant-Owned**  
Information from a source the restaurant controls directly: the restaurant's own website, a PDF menu hosted on the restaurant's domain, or materials provided directly to GoldPan by the restaurant.

**Source Tier 2 — Authorized Third-Party**  
Information from a third-party platform where the restaurant has provided or claimed its own listing: Yelp (owner-claimed), Google Business (owner-verified), Caviar, Toast, or similar platforms where restaurant-provided data is expected.

**Source Tier 3 — Unverified Third-Party or Canvasser Observation**  
Information from sources not controlled or authorized by the restaurant: user-uploaded menus, community-reported information, screenshots from non-official sources, or canvasser direct observation without a documented URL source.

Tier 3 sources require additional scrutiny. Evidence from Tier 3 sources should be tagged with higher uncertainty.

## Lifecycle_Events Schema

The Lifecycle_Events log is append-only. One row is added for every lifecycle status transition. This is the permanent audit trail of the Restaurant_Registry.

| Column | Type | Description |
|--------|------|-------------|
| `Event_ID` | Text | Unique identifier generated by `next_id.py`. Format: `evt_XXXX` |
| `Restaurant_ID` | Text | References the Restaurant_Registry |
| `Event_Date` | Date | Date the transition occurred |
| `From_Status` | Enum | Lifecycle status before the transition |
| `To_Status` | Enum | Lifecycle status after the transition |
| `Actor` | Text | Who made the transition (canvasser name, reviewer name, `pipeline`, or `freshness_system`) |
| `Notes` | Text | What happened or why (required for all suspension, deactivation, and rejection events; optional for standard progressions) |

## Relationship to Other Registries

The Restaurant_Registry is the master record. Other registries reference it.

**Freshness Registry** _(existing)_  
Manages recanvass scheduling and status. The `Recanvass_Status` and `Last_Canvassed` fields in the Restaurant_Registry should mirror or be sourced from the Freshness Registry. If a conflict exists, the Freshness Registry governs freshness decisions; the Restaurant_Registry provides the summary view.

**Menu Source Registry** _(existing)_  
Tracks all source URLs, source tiers, access methods, and source-level status. The `Primary_Source_URL` and `Primary_Source_Tier` in the Restaurant_Registry are summary fields pointing at the restaurant's current primary source. Detailed source records live in the Menu Source Registry.

**GDL — General Dish List** _(existing)_  
Stores all dish and ingredient evidence. The Restaurant_Registry does not contain evidence — it references the restaurant by ID, and the GDL contains all evidence rows keyed to that ID.

## Who Maintains the Registry

| Action | Actor |
|--------|-------|
| Create new prospect record | Canvasser or system (on prospect intake) |
| Approve coverage (`prospect → qualified`) | Canvasser (writes Coverage_Signal, Coverage_Approved fields) |
| Assign canvasser | GoldPan coordinator |
| Update lifecycle status | Canvasser or QA Reviewer (per lifecycle stage) |
| Add Lifecycle_Events row | Required of all actors on every status transition |
| Suspend or deactivate | Requires written justification in Suspension_Reason or Deactivation_Reason |
| Update Evidence_Tier | Canvasser, on completion of confirmation or evidence upgrade |

## Implementation Notes

The Restaurant_Registry and Lifecycle_Events log are database tabs in the GoldPan Google Sheets system. They do not yet exist — they are defined here and should be created before the Restaurant Onboarding subsystem is designed, since onboarding is the first stage that writes to these records.

**When implementing:**
1. Create `Restaurant_Registry` tab with the schema above
2. Create `Lifecycle_Events` tab with the schema above
3. Update `next_id.py` to support `rest_XXXX` and `evt_XXXX` ID sequences
4. Backfill existing restaurants into the Registry with their current lifecycle status
5. Consider whether existing Freshness Registry and Menu Source Registry fields should be migrated, mirrored, or left independent — avoid duplication if possible

## Open Questions

1. **Backfill scope** — How many restaurants are currently in GoldPan? All should be entered into the Restaurant_Registry at their current lifecycle status. This is a one-time migration.
2. **ID format** — `rest_XXXX` is a proposed format. If existing Restaurant_IDs in the GDL already use a different format, the Registry should match them.
3. **Suspension authority** — Who has authority to suspend or deactivate a restaurant? Recommend requiring a second reviewer (not the original canvasser) and mandatory Notes.
4. **Coverage_Signal update on recanvassing** — If a restaurant's dietary signal meaningfully changes between canvassing cycles (e.g., restaurant adds allergen guide), should the Coverage_Signal field be updated to reflect the stronger signal, or should it remain the original justification?
